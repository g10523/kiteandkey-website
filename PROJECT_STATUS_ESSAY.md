# Kite & Key Academy Student LMS — Project Status as of 18 February 2026

---

## A Comprehensive Description of the Current Codebase

The Kite & Key Academy Learning Management System is a React-based, single-page application built for a private tutoring academy. At this stage of development, the project represents a substantial and architecturally ambitious platform that has moved well beyond a minimum viable product. It is a Vite-powered TypeScript application, running React 19 with Framer Motion for animation, Lucide React for iconography, and Axios standing by for eventual server communication. The aesthetic philosophy is deliberate and consistent—a dark-mode, glassmorphic, Japandi-inspired interface that uses soft lavenders, muted purples, and translucent card surfaces to create a learning environment that feels calm, focused, and premium. This is not a generic LMS skin; every page has been designed to communicate pedagogical seriousness combined with visual sophistication.

### The Application Architecture

The application is organised into a clean separation of concerns. The `src/` directory contains approximately ninety files spread across ten subdirectories: `pages/` (twenty-one files across the main views and authentication), `components/` (thirty-four files including specialised sub-folders for assessments, MindPrint visualisations, admin panels, and growth tracking), `services/` (eight files covering API abstraction, course access control, assessment APIs, intervention APIs, MindPrint APIs, client-side scoring, and two persistent stores), `context/` (two React context providers for authentication and data), `types/` (two type definition files totalling over five hundred lines of TypeScript interfaces), `config/` (feature flags), `assessments/` (five files defining cognitive dimensions, scoring engines, test batteries, and assessment services), `interventions/` (two files for cognitive interventions and the intervention engine), and `data/` (mock data exceeding forty-nine thousand bytes, plus a dedicated course file for English Year 5).

The routing system is custom-built using a `PageType` union type that enumerates twenty-two distinct pages. Navigation is handled via a state-driven `handleNavigate` function in `App.tsx`, which sets the current page and passes optional subject, lesson, and dimension identifiers. This is paired with a collapsible sidebar (`Sidebar.tsx`, nearly eight thousand bytes of rendering logic) that provides persistent navigation. A feature flag system (`featureFlags.ts`) gates access to unreleased modules, showing professionally designed "Coming Soon" screens for features not yet ready for student use, including Assignments, Analytics, Messages, Resources, Schedule, and Practice. The Study Lab exists in beta. This approach means the application presents a complete, polished navigation structure even where backend functionality is still pending—students never see a broken or empty page.

### Authentication and Access Control

The authentication layer is implemented through `AuthContext.tsx`, which manages user state, login, registration, and token generation. The system supports four distinct roles—student, parent, tutor, and admin—each defined with their own TypeScript interface extending a base `User` type. Registration is token-gated: administrators generate unique registration codes (formatted as `K&K-ROLE-XXXXXX`) that map to specific roles. A bypass token (`KITE-BETA-2026`) exists for public beta access, but users who register with it are marked as unverified and routed to a `PendingVerification` screen until an admin approves them. This is a thoughtful access control pattern—it allows public discovery of the platform while maintaining administrative gatekeeping over who actually enters the learning environment.

The admin role has access to two critical management surfaces. The first is a Token Management page (`TokenManagement.tsx`) where administrators can generate, view, and delete registration tokens. The second is a comprehensive Admin Panel (`AdminPanel.tsx`, over seventeen thousand bytes) that integrates with a `courseStore` service for managing course access control. This store, persisted entirely in `localStorage`, tracks which courses each user can access, maintains tutor-student pairings with scoped course assignments, and provides a registry of all users for admin selection. The `DataContext` integrates with this store during data loading—when a student logs in, they see only the subjects an administrator has explicitly assigned to them. Administrators, by contrast, see everything. This role-based data filtering flows through `getAccessibleCourseIds()`, which returns `null` for admins (meaning "no filter") and an explicit course ID array for everyone else.

### The MindPrint System — Cognitive Profiling at Scale

The most architecturally distinctive feature of this LMS is MindPrint, a cognitive profiling system that permeates nearly every aspect of the student experience. The system has two parallel type systems: a simplified `MindPrintProfile` in `src/types/index.ts` used by the front-end rendering components, and a more clinically precise `MindPrintProfile` in `src/assessments/types.ts` used by the assessment and intervention engines. The front-end profile includes a learner archetype (`category`: Strategic, Narrative, Procedural, or Creative), five cognitive dimension scores (Conceptual vs. Procedural, Visual vs. Verbal, Speed vs. Depth, Structure Tolerance, and Error Sensitivity), detailed lesson adaptation strategies, feedback interpretation mappings, revision strategies, and long-term growth trajectories.

The assessment engine is far more than a placeholder. It defines eight cognitive dimensions—Working Memory, Processing Speed, Executive Function, Verbal Reasoning, Spatial Reasoning, Pattern Recognition, Focus & Attention, and Cognitive Endurance—each with formal test battery definitions, norm tables, and scoring rubrics. The `cognitiveDimensions.ts` file provides clinical-grade definitions, real-world examples, neurological basis descriptions, and academic impact analyses for each dimension. The `scoringEngine.ts` converts raw assessment scores into percentiles and standard scores. The `assessmentService.ts` manages assessment persistence and profile generation. The `testBattery.ts` defines the actual test items, including digit span (forward and backward), N-back tasks, matrix reasoning, symbol search, and Stroop tests.

This assessment data feeds into the `interventionEngine.ts`, which examines a student's MindPrint profile, identifies development edges (dimensions below the 25th percentile), and automatically assigns evidence-based cognitive interventions. These interventions have defined priorities (critical, recommended, enrichment), expected session counts, and evidence strength ratings. Session logs track fidelity. The entire flow—from assessment administration through scoring, profile generation, and intervention assignment—operates as an integrated pipeline.

The visual rendering of all this data is handled by a dedicated suite of components in `src/components/mindprint/`: `CognitiveRadar.tsx` (a fourteen-thousand-byte radar chart visualisation), `DimensionalDeepDive.tsx` (twelve thousand bytes of detailed dimension exploration), `AssessmentHub.tsx` and `AssessmentRunner.tsx` (the assessment administration interface, totalling thirty-three thousand bytes), `ArchetypeCard.tsx`, `CompositeScores.tsx`, and `OnboardingWelcome.tsx`. The main `MindPrint.tsx` page (nearly eleven thousand bytes) and its companion `MindPrintReport.tsx` (over seventeen thousand bytes) render the student's complete cognitive blueprint—including cognitive signature visualisations, instructional adaptation comparisons, live lesson protocols, feedback interpretation panels, revision strategies, and growth trajectories.

A persistent `MindPrintBadge` component floats on-screen for students, providing constant awareness of their cognitive profile status. The lesson view (`LessonView.tsx`) actively integrates MindPrint data—when a student with a profile loads a lesson, the page displays an "Adaptation Active" banner and applies visual indicators like "Conceptual Chunking Applied" badges to content sections, demonstrating how the same lesson material would be presented differently based on the student's cognitive architecture.

### The Curriculum Data Model

The content hierarchy is cleanly structured: Subjects contain Units, which contain Lessons. Each Lesson has a `LessonContent` object comprising typed sections (explanation, example, practice, reflection), learning objectives, and a "Why It Matters" narrative contextualisation. Subjects track progress, current unit, total and completed units, and a confidence level indicator. Units include a `Checkpoint` system for end-of-unit assessment, with defined passing scores and attempt tracking.

The mock data file is enormous—nearly fifty thousand bytes—and populates three complete subjects for Year 8 (English, Mathematics, and Science) with realistic unit and lesson content. The English Year 5 course has been separated into its own file (`englishYear5.ts`, over seventeen thousand bytes) as the beginnings of a course content management architecture. The lesson content is not placeholder data; it contains actual pedagogical material about character development, plot structure, narrative voice, algebraic expressions, linear equations, cells, atomic structure, and more.

### Weekly Quizzes and Content Visibility Controls

The most recent development work has added a quiz system and content visibility management layer. The type system now includes `Quiz`, `QuizAttempt`, and `ContentVisibility` interfaces. Each lesson can optionally contain a quiz with multiple question types (multiple-choice, true-false, short-answer), configurable time limits, passing scores, attempt limits, and per-quiz visibility controls. Two complete quizzes have been populated into the English curriculum with realistic questions, explanations, and difficulty ratings.

The `QuizView` component (`src/components/QuizView.tsx`, nearly eighteen thousand bytes) provides a full quiz-taking experience: a real-time countdown timer, a question navigator grid showing answered and unanswered status, difficulty indicators, type-specific answer interfaces (radio buttons for multiple choice, large toggle buttons for true/false, text areas for short answers), progress tracking, and a comprehensive results view with question-by-question review, explanations, and retry functionality. The lesson view integrates quizzes with a prominent quiz card at the bottom of content, displaying question count, time limit, passing score, and best score from previous attempts.

The `ContentManagement` component (`src/components/ContentManagement.tsx`, over seventeen thousand bytes) provides a tutor and admin interface for controlling what students see. It includes subject selection, lesson search, status filtering (all, visible, hidden), per-lesson and per-quiz visibility toggles, release date scheduling, and a save mechanism. This component, along with a dedicated quiz page, has been registered as new page types in the `PageType` union and wired into the `App.tsx` routing switch.

### The Backend Layer

A complete Express.js backend exists in the `api/` directory, architecturally documented with ASCII flow diagrams covering the authentication flow (JWT with refresh tokens), the registration token flow, the assessment-to-intervention data pipeline, and a complete RBAC permission matrix. The backend uses Knex for database migration management targeting PostgreSQL, with five migration files covering users, tokens, assessments, interventions, and audit logging. It includes Helmet for security headers, CORS validation, rate limiting (100 requests per 15 minutes general, 5 for auth endpoints), Joi input validation, bcrypt password hashing, and a Docker Compose configuration for containerised deployment. A Postman collection is included for API testing.

However, the frontend currently operates entirely against mock data and `localStorage` persistence. The `ApiService` class in `src/services/api.ts` wraps `localStorage` operations to simulate async API calls, and a parallel `apiClient.ts` exists with Axios configuration ready for the eventual backend connection. The `DataContext` loads data on mount, merges it with locally persisted assessment and intervention data, applies course access filtering, and provides it through React context. The transition from mock data to live backend requires updating the `AuthContext` to use JWT-based authentication and the `DataContext` to use the API client instead of `localStorage`.

### Current State and What Remains

The application is in a state best described as "frontend-complete, backend-staged." The user-facing experience is polished and functional across all implemented features. The development server runs successfully via `npm run dev`, and production builds complete without errors. The MindPrint system—from cognitive assessment through intervention assignment—represents genuine educational technology architecture, not a superficial feature. The quiz system provides authentic assessment capability. The admin panel delivers real access control functionality (persisted in `localStorage`).

What remains is primarily integration work: connecting the React frontend to the Express backend, migrating from `localStorage` persistence to PostgreSQL, enabling live authentication with JWT tokens, and progressively unlocking the gated features (Analytics, Assignments, Messages, Resources, Schedule, Practice). The content management system needs its feature flag entry added and its save mechanism connected to persistent storage. The assessment engine's client-side scoring logic should eventually move server-side for security and consistency. And the quiz system, while fully functional on the frontend, needs backend persistence for attempt tracking across sessions.

The codebase is also carrying some technical housekeeping debt: a few unused import warnings in recently added components, two `showExplanation` and `ContentVisibility` type imports that are declared but not yet consumed, and the feature flags configuration needs entries for the two new page types (`quiz` and `content-management`). None of these are functional blockers—they are the natural artefacts of active, iterative development.

In totality, this is a substantial educational technology platform—approximately two hundred source files across frontend and backend, with a type system spanning over a thousand lines, a cognitive profiling engine grounded in psychometric methodology, a role-based access control system, a quiz engine, content visibility management, and a visual design language that is cohesive and distinctive. It is a project that has been built with deliberate architectural decisions at every layer, positioning it for the transition from a compelling prototype to a production system.
